\chapter{ARQUITECTURA E INTERACCIÓN DEL SISTEMA}

El presente capítulo describe la arquitectura funcional del sistema web propuesto para la administración del Desafío Bebras en Bolivia. La exposición se organiza desde una perspectiva sistémica: primero se presenta la visión general del sistema, posteriormente se identifican los actores que intervienen en el proceso, los módulos que componen la solución y las relaciones funcionales entre ellos. Finalmente, se describe el flujo operativo del concurso y se incorporan representaciones generales que permiten comprender la interacción entre procesos, usuarios y componentes.

La arquitectura planteada responde al objetivo de implementar un sistema modular capaz de sostener el ciclo completo del concurso, desde la comunicación institucional y la inscripción de unidades educativas hasta la ejecución de la prueba, la evaluación de respuestas y la generación de resultados. Esta organización modular permite separar responsabilidades, reducir acoplamientos entre componentes y facilitar la evolución progresiva del sistema conforme cambien los requerimientos operativos de Bebras Bolivia.

\section{VISIÓN GENERAL DEL SISTEMA}

El Sistema Bebras Bolivia se concibe como un sistema web modular orientado a la gestión integral de competencias educativas de pensamiento computacional. Su propósito principal es proporcionar una infraestructura digital que permita organizar el Desafío Bebras en el contexto nacional, considerando tanto las necesidades administrativas de la organización como las condiciones reales de participación de maestros, unidades educativas y estudiantes de nivel primario y secundario.

Desde el punto de vista funcional, el sistema integra cuatro módulos principales: gestión de contenido, gestión de competencias, registro de participantes y evaluación. Estos módulos no operan como aplicaciones aisladas, sino como componentes articulados dentro de un mismo flujo de concurso. La información pública del desafío se administra mediante el CMS y se publica como sitio informativo estático; las competencias y tareas se configuran desde el módulo de gestión académica; las unidades educativas, maestros y estudiantes se organizan en el módulo de participantes; y la resolución de pruebas, junto con la calificación digital o imprimible, se gestiona desde el módulo de evaluación.

La vista de contexto del modelo C4 permite ubicar al Sistema Bebras Bolivia como el sistema central de interacción para la organización y ejecución del desafío. En esta vista se identifican los tres actores principales: administrador u organizador, maestro y estudiante, junto con sus relaciones generales de uso.

Esta separación permite delimitar responsabilidades y niveles de acceso, especialmente porque el acceso al CMS queda reservado exclusivamente al administrador.

La arquitectura tecnológica se apoya en una pila coherente con el marco definido para el proyecto. El panel de administración del CMS se desarrolla con Astro y React, utiliza una base de datos SQLite para usuarios, publicaciones y respaldos, y escribe el contenido editorial en un \textit{Content Store} compuesto por archivos JSON y Markdown. Dicho repositorio de contenido alimenta el proceso de construcción del sitio informativo, implementado con Astro y React como sitio estático generado previamente mediante SSG. La lógica de servidor del concurso se estructura mediante Express y TypeScript, mientras que Prisma ORM y MariaDB proporcionan una capa persistente relacional para competencias, participantes, respuestas y resultados. El servidor web Apache actúa como capa de publicación y acceso HTTP, por lo que sirve el sitio informativo y el frontend del concurso, sin considerarse un contenedor funcional del sistema. Docker se considera como soporte para la estandarización del entorno de ejecución y despliegue.

La vista de contenedores del modelo C4 complementa esta descripción tecnológica al mostrar las principales unidades ejecutables o desplegables del sistema: CMS, base de datos del CMS, \textit{Content Store}, sitio informativo, frontend del concurso, backend del concurso y base de datos del concurso.

El diagrama de contenedores muestra el flujo definitivo de la arquitectura. El CMS utiliza SQLite para almacenar usuarios del panel, publicaciones y respaldos, y escribe los archivos editoriales en el \textit{Content Store} basado en JSON y Markdown. Dicho almacén funciona como fuente del proceso de construcción del sitio informativo estático. Para la operación del concurso, el frontend consume por API REST el backend desarrollado con Express y TypeScript, que implementa la lógica de competencias, registro de participantes y evaluación, y persiste sus datos en MariaDB mediante Prisma.

Esta visión permite que el sistema funcione como un entorno de concurso en línea y no únicamente como un sitio informativo. En consecuencia, la arquitectura debe responder a requisitos de autenticación, registro, configuración de pruebas, control de tiempos, almacenamiento de respuestas, generación de resultados y soporte a escenarios con conectividad limitada. La modularidad constituye, por tanto, una decisión central de diseño, ya que permite atender cada necesidad sin comprometer la coherencia global del sistema.

\section{ESTRUCTURA FUNCIONAL DEL SISTEMA}

La estructura funcional del sistema se organiza en torno a los actores que participan en el Desafío Bebras y a los módulos que materializan sus principales capacidades. Esta división facilita la identificación de responsabilidades, permisos y flujos de interacción, además de proporcionar una base clara para el desarrollo incremental de la solución.

\subsection{Actores del sistema}

Los actores del sistema representan los perfiles de usuario que interactúan con el sistema durante las distintas etapas del concurso. Cada actor posee objetivos, responsabilidades y niveles de acceso diferenciados, por lo que la arquitectura funcional debe contemplar interfaces y permisos acordes con su rol.

\begin{itemize}
    \item \textbf{Administrador u organizador:} responsable de la configuración global del sistema y del concurso. Sus funciones comprenden la administración del contenido institucional, la creación y publicación de competencias, la gestión del banco de tareas, la supervisión de inscripciones, el seguimiento de la ejecución del concurso y la generación de resultados generales. Este actor requiere acceso a los módulos administrativos y a la información consolidada del sistema.

    \item \textbf{Maestro:} responsable operativo dentro de una unidad educativa. Su participación se centra en la inscripción o gestión de grupos de estudiantes, la distribución de códigos o enlaces de acceso, el acompañamiento durante la ejecución de la prueba y la consulta de resultados correspondientes a sus participantes. Su interacción con el sistema debe ser directa y guiada, evitando configuraciones complejas que no correspondan a su función.

    \item \textbf{Estudiante:} participante principal del desafío. Su interacción con el sistema se concentra en el acceso a la prueba, la lectura de tareas, la resolución de ejercicios y el envío de respuestas. Dado que este actor puede pertenecer a distintos niveles escolares y poseer grados variables de experiencia digital, la interfaz destinada a estudiantes debe priorizar claridad, simplicidad, continuidad de sesión y reducción de carga cognitiva.
\end{itemize}

La delimitación de estos actores permite establecer una arquitectura orientada por responsabilidades. El administrador organiza y controla el concurso; el maestro articula la participación de su grupo; y el estudiante ejecuta la prueba. Esta separación contribuye a reducir errores operativos y a mantener flujos de interacción adecuados para cada perfil.

\subsection{Articulación funcional de los módulos}

La articulación funcional de los módulos se produce mediante un flujo de información progresivo. En primer lugar, el CMS permite al administrador editar la información institucional que comunica el calendario, los requisitos y las orientaciones del desafío. El contenido se guarda en el \textit{Content Store} y alimenta el sitio informativo estático, publicado mediante Apache. Esta información permite que maestros e instituciones conozcan el proceso de participación y accedan a los canales de inscripción definidos por la organización.

En segundo lugar, el módulo de gestión de competencias define la estructura académica del concurso. En esta etapa se registran las competencias, categorías, niveles, duración, tareas y reglas de calificación. Este módulo funciona como el núcleo de configuración del desafío, ya que determina qué será resuelto por los estudiantes y bajo qué condiciones.

En tercer lugar, el módulo de registro de participantes vincula las competencias configuradas con las unidades educativas, maestros y estudiantes inscritos. A partir de esta relación, el sistema puede conformar grupos, generar mecanismos de acceso y preparar la información necesaria para la ejecución de la prueba.

Finalmente, el módulo de evaluación utiliza la información previamente configurada para presentar las tareas a los estudiantes, registrar sus respuestas y calcular resultados. En los casos en que la conectividad sea limitada, el sistema contempla la generación de formatos imprimibles como mecanismo complementario, permitiendo que la evaluación no dependa exclusivamente de una sesión digital continua.

La integración entre módulos se basa en una lógica de dependencia funcional. La evaluación requiere competencias publicadas y participantes registrados; el registro requiere una convocatoria o competencia disponible; y la comunicación pública se alimenta de la planificación institucional del concurso. De esta forma, el sistema mantiene un flujo coherente entre comunicación, configuración, inscripción, ejecución y resultados.

\section{FLUJO OPERATIVO DEL DESAFÍO}

El flujo operativo del Desafío Bebras describe la forma en que el sistema acompaña el proceso completo del concurso. Este flujo integra actividades previas, actividades durante la ejecución y actividades posteriores a la evaluación, de modo que el sistema pueda sostener tanto la preparación administrativa como la experiencia de participación de los estudiantes.

\subsection{Proceso general de funcionamiento}

El funcionamiento general del sistema inicia con la preparación institucional del desafío. En esta etapa, el administrador actualiza el contenido desde el CMS, publica la convocatoria, incorpora recursos de orientación y habilita la información necesaria para maestros e instituciones. El CMS guarda los cambios en el \textit{Content Store}, desde donde se construye el sitio informativo estático servido por Apache. Esta fase cumple una función comunicacional, ya que establece las condiciones de participación y facilita la difusión del concurso.

Posteriormente, el administrador configura la competencia en el módulo correspondiente. Esta configuración incluye la definición de categorías, niveles, fechas, duración, tareas y criterios de evaluación. La competencia se considera operativamente disponible cuando su estructura ha sido validada y publicada para su uso por maestros o unidades educativas.

Una vez publicada la competencia, el proceso continúa con el registro de participantes. Los maestros o responsables institucionales inscriben grupos y estudiantes, o administran la información proporcionada por la organización. A partir de estos datos, el sistema habilita mecanismos de acceso que permiten a los estudiantes ingresar a la evaluación de manera controlada.

Durante la ejecución del concurso, los estudiantes acceden a la prueba, resuelven las tareas asignadas y envían sus respuestas. El sistema registra el avance de la sesión, controla el tiempo disponible y almacena las respuestas para su posterior calificación. Cuando la prueba concluye, ya sea por entrega voluntaria o por finalización del tiempo, el módulo de evaluación procesa los resultados conforme a las reglas definidas en la competencia.

La etapa final comprende la revisión, consolidación y consulta de resultados. El administrador puede obtener información global del concurso, mientras que los maestros acceden a los resultados de sus grupos o estudiantes. Esta información permite cerrar el ciclo operativo del desafío y generar insumos para reportes institucionales, reconocimientos o análisis posteriores.

\subsection{Secuencia operativa del concurso}

La secuencia operativa del concurso organiza el proceso en pasos consecutivos que representan el recorrido funcional esperado del sistema. Aunque algunos pasos pueden variar según la modalidad de participación o las decisiones organizativas de cada edición, la secuencia general se mantiene estable.

\begin{enumerate}
    \item El administrador actualiza el contenido público del sitio, incluyendo convocatoria, fechas, orientaciones y recursos de apoyo para maestros e instituciones.
    \item El administrador crea o actualiza la competencia, define categorías, niveles, duración y condiciones generales de participación.
    \item El administrador selecciona o registra tareas de pensamiento computacional y las asocia a la competencia según edad, dificultad y criterios de evaluación.
    \item La competencia se publica y queda disponible para el proceso de inscripción o asignación de grupos.
    \item El maestro o responsable institucional registra la unidad educativa, conforma grupos e inscribe a los estudiantes participantes.
    \item El sistema genera o habilita los accesos necesarios para que los estudiantes puedan ingresar a la prueba correspondiente.
    \item El estudiante accede a la evaluación, revisa las tareas asignadas y registra sus respuestas dentro del tiempo establecido.
    \item El sistema almacena las respuestas y, cuando corresponde, conserva el avance para permitir continuidad ante interrupciones controladas.
    \item Al finalizar la prueba, el sistema procesa la evaluación mediante reglas de calificación definidas para la competencia.
    \item El administrador revisa resultados generales, mientras que el maestro consulta los resultados asociados a sus estudiantes o grupos.
\end{enumerate}

Esta secuencia expresa un flujo lineal de referencia, pero la arquitectura modular permite que determinadas actividades se realicen de manera paralela. Por ejemplo, la preparación de contenido institucional puede continuar mientras se ajusta el banco de tareas, y el registro de participantes puede desarrollarse mientras la organización revisa aspectos operativos de la competencia. La separación entre comunicación pública, administración de contenido y operación del concurso facilita esta flexibilidad sin perder trazabilidad ni comprometer el funcionamiento general del sistema.
