\chapter{MÓDULO DE GESTIÓN DE CONTENIDO Y SITIO INFORMATIVO}

El módulo de gestión de contenido y sitio informativo constituye el componente encargado de administrar la presencia pública de Bebras Bolivia. Su alcance no se limita al panel CMS, sino que incluye también el sitio informativo mediante el cual visitantes, estudiantes, maestros, coordinadores e instituciones pueden consultar información oficial sobre el desafío.

El módulo integra dos dimensiones complementarias. La primera corresponde al sitio público, compuesto por páginas informativas, navegación, contenido institucional, preguntas frecuentes, información para estudiantes y maestros, patrocinadores, contacto, registro informativo y publicaciones. La segunda corresponde al panel administrativo, desde el cual se puede editar contenido, administrar publicaciones, manejar imágenes, previsualizar cambios, crear respaldos, restaurar contenido y publicar el sitio.

El desarrollo se organizó mediante iteraciones XP de tres semanas. La Iteración 0 se destinó a la preparación funcional y técnica del módulo; la Iteración 1 concentró el sitio informativo, el acceso administrativo, la edición de páginas y la vista previa del sitio; y la Iteración 2 consolidó la gestión editorial, el manejo de imágenes, los respaldos y la publicación del contenido.

\section{PLANIFICACIÓN DEL MÓDULO}

\subsection{Objetivo del módulo}

El objetivo del módulo es permitir la comunicación pública del Desafío Bebras Bolivia y facilitar la administración del contenido del sitio sin modificar directamente el código fuente. De esta manera, el sistema permite mantener actualizada la información institucional, las páginas informativas, las publicaciones y los recursos visuales asociados al desafío.

    \item Se preparó la base técnica necesaria para continuar con el desarrollo incremental en las siguientes iteraciones.
\end{itemize}

\section{ITERACIÓN 1}

La primera iteración del módulo se enfocó en construir la base visible y editable del sitio informativo. Bajo la lógica XP, se priorizó entregar software funcional en un plazo corto: un visitante debía poder consultar la información principal del desafío y un administrador debía poder ingresar al panel, modificar contenido estructurado y revisar el resultado antes de publicarlo.

\subsection{Planificación de la iteración}

La planificación de la Iteración 1 consideró cuatro historias principales. La historia CMS-01 agrupó la presentación pública del desafío y se descompuso en tareas relativas a información general, categorías por edad, contenido para estudiantes, contenido para maestros, puntuación, preguntas frecuentes, patrocinadores, contacto, estado de registro y navegación. La historia CMS-02 cubrió el control de acceso al panel administrativo. La historia CMS-03 abordó la edición de páginas informativas y se apoyó en tareas de validación de estructura, administración de secciones reutilizables y ordenamiento de componentes. La historia CMS-04 incorporó la vista previa de cambios antes de su publicación.

\subsection{Diseño}

El diseño de la Iteración 1 se centró en separar el contenido editable de la presentación visual del sitio. Esta decisión permitió que el sitio informativo funcionara como una landing pública generada a partir de contenido estructurado, mientras que el panel CMS quedó encargado de modificar, organizar y previsualizar dicho contenido antes de su publicación.

Desde el enfoque XP, el diseño se mantuvo simple e incremental. No se incorporó una arquitectura compleja de gestión de contenido desde el inicio; se definió un flujo directo entre el administrador, el panel CMS, el almacén de contenido y el sitio público. Esta estructura permitió entregar valor en la iteración sin acoplar el contenido a las páginas del sitio.

\subsubsection{Arquitectura de contenido editable}

La primera decisión de diseño fue tratar el contenido del sitio como información estructurada y no como texto fijo dentro de las páginas. El administrador modifica el contenido desde el panel CMS; el sistema guarda esos cambios en un almacén de contenido editable y luego sincroniza una versión consumible por la landing pública. De esta forma, el sitio puede mostrar información actualizada sin que el administrador tenga que modificar directamente la implementación visual.

\input{content/graficos_cms/cms_contenido_editable_i1}

Este diseño responde principalmente a las historias CMS-01 y CMS-03. Para el visitante, el resultado es un sitio informativo con páginas públicas organizadas por secciones. Para el administrador, el resultado es un mecanismo de edición que actúa sobre el contenido estructurado y no sobre el código del sitio.

\subsubsection{Composición dinámica de páginas}

La segunda decisión de diseño fue permitir que las páginas se compongan mediante bloques reutilizables. Cada página define una secuencia de secciones, y cada sección indica el tipo de bloque que debe mostrarse. El renderizador de páginas interpreta esa composición y construye la vista final utilizando componentes visuales predefinidos.

\input{content/graficos_cms/cms_composicion_paginas_i1}

Esta composición permitió cubrir varias secciones del sitio sin crear una solución distinta para cada página. En lugar de duplicar estructuras visuales, el sistema reutiliza bloques comunes para información general, listas, preguntas frecuentes, formularios, categorías y tablas. Esto mantiene el diseño simple, facilita la edición desde el panel y permite que nuevas secciones puedan agregarse de manera incremental.

\subsubsection{Flujo de vista previa}

La tercera decisión de diseño fue incorporar una vista previa antes de publicar cambios. El objetivo fue reducir el ciclo de retroalimentación del administrador: después de modificar contenido, el sistema puede preparar una versión temporal y mostrarla dentro del panel sin afectar la versión pública definitiva.

\input{content/graficos_cms/cms_vista_previa_i1}

Este flujo responde a la historia CMS-04. La vista previa permite revisar el efecto de una modificación en el contexto real del sitio, manteniendo separada la edición del contenido de la publicación definitiva. Así, el administrador puede detectar errores de redacción, orden o composición antes de hacer visible el cambio a los visitantes.

\subsection{Desarrollo}

El desarrollo de la Iteración 1 materializó la arquitectura definida en el diseño. En esta sección se describe cómo se organizó e implementó el módulo, sin detallar cada archivo de código fuente, sino agrupando las responsabilidades por unidades de implementación.

\subsubsection{Organización del código fuente}

La implementación se organizó en unidades diferenciadas para mantener separado el sitio público, el panel administrativo, los servicios del CMS y los scripts de apoyo. Esta organización permitió trabajar de forma incremental sobre cada historia de usuario, manteniendo una correspondencia clara entre funcionalidad y responsabilidad técnica.

\subsubsection{Desarrollo del sitio informativo}

Para la historia CMS-01 se implementó la base pública del sitio informativo. El sitio fue construido como una landing compuesta por páginas y secciones reutilizables, de modo que la información principal del Desafío Bebras Bolivia pudiera presentarse de forma ordenada. La implementación incluyó contenido general del desafío, categorías de participación por edad, secciones para estudiantes, secciones para maestros y coordinadores, sistema de puntuación, preguntas frecuentes, patrocinadores, contacto institucional, estado informativo del registro y navegación entre páginas.

El desarrollo del sitio público se apoyó en un modelo de contenido estructurado. En lugar de escribir cada texto directamente dentro de las vistas, las páginas reciben datos organizados y los transforman en bloques visuales. Esta decisión reduce duplicación, facilita la actualización posterior del contenido y permite que el CMS pueda modificar información sin alterar la presentación base del sitio.

También se implementó una estructura visual común para las páginas públicas. Esta estructura incluye cabecera, navegación, contenido principal, pie de página, metadatos del sitio y soporte para dispositivos móviles. Con ello se aseguró que las diferentes páginas mantuvieran una identidad visual consistente y que el visitante pudiera recorrer el sitio desde distintos tamaños de pantalla.

\subsubsection{Desarrollo del panel administrativo}

Para la historia CMS-02 se desarrolló el acceso protegido al panel administrativo. El panel solicita credenciales al administrador y el backend valida los datos antes de permitir el ingreso. Cuando las credenciales son correctas, el sistema crea una sesión que habilita el acceso a las funciones internas del CMS; cuando son incorrectas, se informa el error y se mantiene restringido el acceso.

El panel administrativo se implementó como una interfaz de trabajo para gestionar el contenido del sitio. La navegación interna organiza las secciones editables y separa el contenido informativo de otras funciones del sistema. Esta organización permitió que el administrador ubique rápidamente la página o sección que desea modificar, sin tener que conocer la estructura técnica del proyecto.

Además del ingreso, se implementó la protección de rutas administrativas y la opción de cerrar sesión. Con ello, las operaciones de edición, vista previa y guardado quedaron reservadas para usuarios autenticados, cumpliendo el objetivo de proteger la administración del contenido público.

\subsubsection{Desarrollo del editor de contenido}

Para la historia CMS-03 se desarrolló un editor dinámico capaz de trabajar con contenido estructurado. El editor no fue construido como un formulario fijo para una sola página; en su lugar, interpreta la estructura del contenido y genera controles de edición según el tipo de dato. Esto permite editar textos, enlaces, valores booleanos, listas, secciones anidadas y componentes reutilizables.

El editor también incorporó operaciones para administrar secciones dentro de una página. El administrador puede agregar nuevos elementos, eliminar secciones que ya no son necesarias y modificar el orden de los bloques. Esta capacidad es importante porque el sitio no solo requiere cambiar textos, sino también reorganizar la manera en que se presenta la información pública.

La edición se diseñó para mantener una experiencia simple. Los campos principales se muestran de forma directa, mientras que los detalles de diseño o configuración avanzada se mantienen separados para no sobrecargar al administrador. De esta manera, el panel prioriza la edición del contenido visible y conserva la flexibilidad necesaria para ajustar la composición de las páginas.

\subsubsection{Desarrollo de la vista previa}

Para la historia CMS-04 se implementó el flujo de vista previa. Esta funcionalidad permite revisar los cambios del sitio antes de publicarlos definitivamente. Cuando el administrador inicia la vista previa, el CMS prepara una versión temporal del contenido y la muestra dentro del propio panel administrativo.

Durante la edición, los cambios pueden enviarse como borrador hacia el entorno de previsualización. Para evitar actualizaciones excesivas mientras el administrador escribe, se utilizó una actualización diferida: el sistema espera un breve intervalo y luego sincroniza el borrador. Esta decisión mejora la experiencia de edición, porque permite observar cambios recientes sin obligar al usuario a guardar o publicar cada modificación.

La vista previa fue desarrollada como un mecanismo separado de la publicación. Esto significa que el administrador puede verificar el contenido en el contexto real del sitio, pero la versión pública no se reemplaza hasta que se realice una acción de publicación en una iteración posterior. Esta separación reduce el riesgo de publicar errores de redacción, orden o composición.

\subsection{Diseño}

El diseño de la Iteración 2 amplió la base construida en la Iteración 1. La arquitectura no se reemplazó, sino que se extendió para cubrir el ciclo editorial completo: creación de publicaciones, manejo de imágenes, respaldos, restauración, transferencia de copias y publicación del sitio. Esta ampliación exigió diferenciar dos tipos de persistencia: el contenido editable que alimenta al sitio público y los registros operativos que controlan respaldos, publicaciones y programación.

En esta iteración, el almacén de contenido se mantuvo como fuente publicable en archivos estructurados, principalmente JSON para páginas informativas y Markdown para publicaciones. En cambio, SQLite se utilizó para datos administrativos del CMS, como usuarios, metadatos de respaldos, historial de publicación y publicaciones programadas. Esta separación evitó mezclar contenido visible para visitantes con información interna de operación.

Las publicaciones informativas se diseñaron como contenido dinámico del sitio, pero sin depender de una base de datos para su renderizado público. Cada publicación mantiene metadatos, cuerpo de contenido enriquecido e imagen opcional. El CMS valida esos datos, los guarda en el almacén editorial y permite previsualizarlos antes de que formen parte de la versión publicada del sitio.

\subsubsection{Diseño del ciclo de respaldos}

Los respaldos se diseñaron como copias completas del estado editorial del contenido. Un respaldo no solo registra una fila administrativa, sino que conserva archivos de contenido y publicaciones en una carpeta versionada. SQLite mantiene los metadatos necesarios para listar, ordenar y ubicar cada respaldo; los archivos conservan el contenido recuperable.

Cuando el administrador confirma la publicación, se crea un respaldo automático, se copian los archivos publicables y se ejecuta la construcción del sitio. \subsection{Desarrollo}

El desarrollo de la Iteración 2 se concentró en convertir el CMS en una herramienta editorial completa. A diferencia de la Iteración 1, donde el esfuerzo principal fue habilitar la edición de páginas informativas, esta etapa incorporó operaciones de ciclo de vida: crear publicaciones, administrar imágenes, conservar respaldos, restaurar estados anteriores y controlar la publicación del sitio.

\subsubsection{Implementación de publicaciones informativas}

Para las publicaciones, se implementó un flujo basado en metadatos y contenido enriquecido. El panel administrativo permite registrar título, descripción, fecha, autor, imagen y cuerpo de la publicación. Antes de guardar, el servicio valida la estructura del contenido para evitar publicaciones incompletas. La vista previa editorial permite revisar el resultado sin modificar todavía la versión publicada del sitio.

La implementación separó la información de presentación del contenido editable. Los metadatos permiten ordenar y describir la publicación dentro del sitio, mientras que el cuerpo enriquecido mantiene el contenido principal. Esta decisión facilita que el listado de publicaciones, la página de detalle y la vista previa usen la misma fuente de datos sin duplicar información.

\subsubsection{Implementación del manejo de imágenes}

El manejo de imágenes se implementó como un servicio separado del contenido textual. Esta decisión permitió validar extensiones, limitar archivos permitidos y generar rutas utilizables desde las publicaciones. De esta forma, las imágenes se mantienen como recursos estáticos y las publicaciones solo conservan referencias hacia ellas.

Durante la carga se controlan los tipos de archivo admitidos para evitar formatos inseguros o incompatibles con el uso previsto del sitio. El resultado de la carga no se almacena como contenido embebido dentro de la publicación, sino como una ruta reutilizable. Esto reduce el acoplamiento entre publicaciones e imágenes y permite que el proceso de publicación copie los recursos visuales junto con el contenido textual.

\subsubsection{Implementación de respaldos y restauración}

En respaldos, el desarrollo combinó persistencia en archivos y registro operativo. Los archivos JSON y Markdown se copian a una carpeta de respaldo versionada, mientras que SQLite registra autor, descripción, fecha y ubicación del respaldo. Esta combinación permite listar respaldos desde el panel y, al mismo tiempo, conservar una copia recuperable del contenido.

La restauración se desarrolló como la operación inversa al respaldo: el contenido guardado en una copia seleccionada reemplaza el contenido actual de trabajo. Para reducir errores, los respaldos mantienen una estructura diferenciada entre datos de páginas y publicaciones. Además, la transferencia de respaldos se implementó mediante archivos comprimidos, de modo que una copia pueda descargarse, conservarse externamente e importarse nuevamente cuando sea necesario.

\subsubsection{Implementación de publicación y programación}

La publicación del sitio se implementó como un proceso controlado por el CMS. El sistema identifica diferencias entre el contenido actual y el contenido publicado, crea un respaldo automático antes de publicar, copia datos, publicaciones e imágenes hacia el sitio público y ejecuta la construcción del proyecto. El resultado de la operación queda registrado para informar si la publicación fue exitosa o fallida.

La comparación previa cumple una función operativa importante: permite que el administrador revise qué archivos o secciones cambiarán antes de actualizar el sitio visible. La publicación programada reutiliza el mismo proceso, pero agrega una capa de control temporal mediante SQLite. Allí se registra la fecha de ejecución, el autor y el estado de la programación, lo que permite cancelar una publicación pendiente o consultar el historial reciente.
